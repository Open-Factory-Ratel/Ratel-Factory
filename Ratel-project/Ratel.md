# AI Software Factory Architecture & Strategy

## The Architecture
To build an effective AI software factory, the system must move beyond basic one-on-one chat interfaces into a composed, multi-agent ecosystem. Based on the sources, the architecture should be built upon the following pillars:

*   **Custom Agent Harness:** You must own and control the environment (harness) where your agents operate, rather than relying on out-of-the-box defaults. This allows you to compose unique workflows, specialized domain agents (e.g., DevOps, testing), and custom fallback routing.
*   **Three-Role Multi-Agent System (Missions):** 
    *   **Orchestrators:** Handle the planning phase. They act as a sounding board, ask strategic questions, and generate a "validation contract" that defines exactly what a successful feature looks like before any code is written.
    *   **Workers:** Handle execution. They receive a clean slate of context, read the specification, implement the feature, and commit the code via git. 
    *   **Validators:** Handle adversarial verification. A *scrutiny validator* runs tests, linters, and code reviews, while a *user testing validator* acts like a QA engineer, navigating the live application visually to ensure end-to-end functionality.
*   **Structured Handoffs & State:** Instead of hoping agents remember context, workers must generate structured handoffs detailing what was completed, exit codes, and issues found. This shared state allows the system to course-correct at milestone boundaries.
*   **Serial Execution with Read-Only Parallelization:** To avoid agents overwriting each other's work or making conflicting architectural decisions, execution should happen serially. Parallel processing should only be used for read-only operations like codebase search or code review.
*   **Extensible Codebase & Agentic Access:** The software must be highly pluggable and adaptable ("open to extension, closed to modification") so the factory can survive rapid model changes. Furthermore, agents must be given direct access to CLIs and APIs to operate at "agentic speed" without relying on manual workarounds.

## Why We Need the Software Factory
*   **Overcoming the Human Attention Bottleneck:** The primary limitation in software engineering is no longer AI intelligence; it is human bandwidth. A human engineer can only review and push a few features forward per day, even if an AI is smart enough to code 50 of them.
*   **Escaping "Vibe Coding":** Manually prompting models back-and-forth in a terminal is the lowest baseline of AI engineering. Instead of building single features yourself, you must build "the system that builds the system" to achieve actual leverage.
*   **Guaranteeing On-Spec, Reproducible Results:** A factory workflow ensures that complex engineering work is templated. This yields repeatable, high-quality outcomes every single time, rather than relying on the varied success of individual prompting sessions.
*   **Token Arbitrage:** To succeed economically, you must graduate from simply burning tokens ("token maxing") to creating useful tokens, and finally capturing the revenue or business value generated by that AI labor. 

## What's the Expectation From This Software Factory?
*   **Parabolic Productivity Increases:** By shifting focus to factory workflows, your output per unit of time scales exponentially. For example, a team of five engineers currently managing 10 work streams could easily manage 30 concurrent work streams.
*   **Always-On (AFK) Agents:** The factory is expected to run asynchronously for hours or even days without human intervention. Engineers will define the scope, approve a plan, and step away while the factory works around the clock.
*   **Self-Healing and Drift Prevention:** Through rigorous validation contracts, the factory will catch bugs organically. The system will independently identify when it fails, scope out corrective work, and pull itself back on track without human steering.
*   **Cleaner Codebases:** Surprisingly, the rigorous structure required by the factory (end-to-end tests, unit tests, explicit skills) leaves the resulting codebase cleaner and more easily navigable for both humans and agents than it was initially.
*   **Zero Touch Engineering (ZTE):** Ultimately, the expectation is to progress toward a state where a high-level prompt can be translated by the factory directly into near-production or fully production-ready systems.
